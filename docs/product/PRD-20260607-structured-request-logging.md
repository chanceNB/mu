# PRD - 结构化请求日志

状态：已完成（2026-06-07）

## 1. 背景

后端已经具备 `X-Trace-Id`、统一异常响应、Agent Trace、Model Call Log 和 Token Usage Log，但普通 HTTP 请求缺少统一的请求完成日志。当前排查线上问题时，只能从业务表或测试响应中拼接上下文，不能稳定看到一次请求的 `traceId`、用户、路由、状态、耗时和错误码。

P3-5 可观测性待办要求补齐结构化日志。本切片选择最小可交付项：请求完成时输出一行白名单字段日志，不引入日志依赖，不改 API，不改数据库。

## 2. 目标

- 为每个后端 HTTP 请求输出结构化访问日志。
- 日志字段限定为 `traceId`、`userId`、`route`、`status`、`latencyMs`、`errorCode`。
- 成功请求记录 `errorCode=OK`。
- 由统一异常处理映射的错误请求记录真实 `ErrorCode`。
- 日志不包含请求体、响应体、敏感 Header、完整 query string、Prompt、RAG 原文或模型输出。

## 3. 范围

纳入：

- `common` 横切层新增请求日志过滤器。
- `GlobalExceptionHandler` 写入内部请求属性，供日志过滤器读取 `errorCode`。
- 请求日志测试覆盖成功、业务异常、校验异常、未知异常和用户上下文。
- 更新 P3-5 TODO、证据、验收、记忆、变更日志和复盘。

不纳入：

- Micrometer 指标。
- 健康检查深度探测。
- 慢查询、慢模型调用、无引用回答、审核积压告警。
- JSON log encoder、日志采集平台或外部可观测性 SDK。
- API 响应结构变更。
- 数据库 schema 变更。

## 4. 用户价值

- 运维人员能用 `traceId` 快速关联一次 HTTP 请求和后端内部 Agent/RAG 证据。
- 后端开发能定位高延迟请求和错误码分布，减少跨模块排查成本。
- 安全审计能确认日志只有白名单字段，不把学习内容、Prompt 或凭据写入日志。

## 5. 成功标准

- 请求完成日志稳定包含六个白名单字段。
- `X-User-Id` 对应用户在日志中可见，不因过滤器清理顺序退化为默认用户。
- `ApiException`、校验异常、未知异常日志能记录对应 `errorCode`。
- 未知异常日志不包含原始异常消息。
- 不新增依赖、不改 API、不改 DB。
